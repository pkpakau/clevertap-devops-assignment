Task 1c: Design Question
------------------------
1. Create a separate EU account with cluster and managed by single argocd control plane
2. No Transit Gateway or peering to US/APAC as blocked by Data residency EU Law no transit/rest in another region (bound to eu-west-1)
3. Add IAM permission Boundary restricting only eu-west-1 region 
4. For CI/CD Deployment, create separate GitHub environment with region specific configs
5. Manage a central argocd in the control plane and creates ApplicationSet and other accounts with their deployments as argo targets as new application (think of hub and spoke)
6. Advantages: all app are managed by single argocd with different helm values, consistent architecture
7. Can Setup AWS Config Rules if any resource is present in another region, then mark the resource non complaint and alert them to team.
8. Audit logs to same account

Section 4: Cost Engineering 

Task 4a: 
Based on my past experience and our workload I think 

Week 1-2 - Quick Wins
----------------------
1. If we are not using IaC like terraform or CDK, lots of resources gets untracked especially Elastic IPs - delete this [ Low Effort | Medium Risk ]
2. Check for any stale EBS volumes as this stays when we delete EC2s as it is not managed by IaC default behavior is to retain the EBS volumes [ Low Effort | High Risk ]
3. Check Stale EC2s, I've noticed manually created EC2s mostly don't have tags attached ( flag for detecting such nodes )
4. Check Cost Explorer for most expensive services and clearing up loose infras and chase stakeholders or teams to clear up resources or intimidation for future deletion of this resources


Week 3-6
--------
1. Don't create separate NAT Gateways for each public subnet in non PROD or UAT envs ( NATs are expensive and for multi vpc environment it becomes major factor of costs 32$/NAT/month)
[ Medium Effort | Medium Risk - requires discussions ] 
2. S3 Tiering would be helpful in storing vpc flow logs which might be needed in ISO audits
[ Medium Effort | Low Risk ]
3. Re-architecture needed 
	- Use spot instances in case of dev, testing envs
	- on demand for fixed workloads
	- Reserved instance to get maximum discounts for predictable workloads (upfront payment needed but pre-validation of workflow needed)
[ Medium Effort | High Risk - effects workload stress tests ]


Long Term
----------
1. Check DBs CPU and Memory Utilization of RDS or Postgres, Scale down if you can
2. Move to Managed Postgres ( Aurora Postgres ) if auto scaling is operationally overhead (yes, it expensive but saves a lot of time and have configurable ACUs and can scale down to 0 too
3. If we are accessing s3 bucket from private subnet resources, create VPC Endpoint Gateway as it has low latency, free same region data transfer rather than traditional public internet way
4. Use AWS Graviton CPUs to get cheaper instances plus faster

Mostly I see silent network costs and its most non obvious. Other issues can be flagged once I see the internal architecture.
